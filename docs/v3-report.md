# V3 — Redis Queue (대기열 시스템) 부하 테스트 리포트

## TL;DR

**100석 콘서트에 500건 동시 예매 → 전부 대기열 진입 성공(HTTP 0% 실패), 100건 예매 확정, 400건 정상 거절.** Enqueue 응답시간 p95=19ms. DB에는 배치 5건씩만 접근하여 커넥션 풀 고갈 없음. 사용자 경험(즉시 응답) + DB 보호를 동시에 달성.

---

## 테스트 환경

| 항목 | 값 |
|------|-----|
| Backend | Spring Boot 3.4.4, Java 21, HikariCP (pool=10) |
| DB | MySQL 8.0 (Docker, port 3307) |
| Redis | Redis 7.0 (Docker, port 6379) |
| 부하 도구 | k6 v0.56.0 |
| VUs | 50 동시 사용자 |
| Iterations | 500 요청 |
| 총 좌석 | 100석 |
| 대기열 | Redis Sorted Set (score = timestamp) |
| 스케줄러 | 200ms 간격, 배치 5건 |

## 테스트 시나리오

```
50명의 사용자가 동시에 랜덤 좌석(1~100)으로 예매 요청
→ Redis 대기열에 enqueue (즉시 응답)
→ Scheduler가 200ms마다 5건씩 dequeue → DB 예매
→ 클라이언트는 polling으로 결과 확인
```

| 지표 | V2 (Lock) | V3 Enqueue | V3 Total (polling 포함) |
|------|-----------|------------|------------------------|

| Max | 658ms | **26.3ms** | **2,955ms** |

> **Total 시간(2.5초)**: polling 포함한 전체 결과 확인 시간 (스케줄러 배치 처리 대기)

### 핵심 설정값과 산출 근거

```
Scheduler Interval = 200ms
Batch Size = 5

이론적 처리량 = 5 / 0.2 = 25 req/s
실제 관측 처리량 ≈ 20 req/s (DB 트랜잭션 오버헤드 포함)

100석 완판 소요시간 = 100 / 20 ≈ 5초

Little's Law (스케줄러 관점):
λ = 25 req/s (배치 도착률)
W = ~50ms (개별 트랜잭션 시간, 락 경합 최소)
L = 25 × 0.05 = 1.25 (동시 DB 트랜잭션)
→ 커넥션 풀 10개 중 1~2개만 사용 = 여유 충분
```

### Batch Size 산출 공식

```
Batch Size = (Pool Size / W_db) × Interval × Safety Factor

= (10 / 0.05) × 0.2 × 0.7
= 200 × 0.2 × 0.7
= 28

실제 적용: 5 (보수적으로 설정)
이유: DB 부하 최소화 + 다른 API도 커넥션 필요
```

## V2 vs V3: 결정적 차이

### DB 커넥션 사용 패턴

**V2 (비관적 락)**
```
[t=0ms]  50 VUs → 10 커넥션 획득 + 40 대기
[t=100ms] 락 해제 → 다음 10개 획득 → 40 계속 대기
[t=200ms] ...반복...
→ 커넥션 풀 전체가 항상 100% 사용
→ connection-timeout 위험
```

**V3 (Redis 대기열)**
```
[t=0ms]  50 VUs → Redis enqueue (커넥션 불필요)
[t=200ms] Scheduler: 5건 dequeue → 5 커넥션 사용
[t=400ms] Scheduler: 5건 dequeue → 5 커넥션 사용
→ 커넥션 풀 50% 이하 사용
→ connection-timeout 불가능
```

### 사용자 경험

**V2**: 예매 버튼 클릭 → 131ms~658ms 대기 → 성공/실패 (80%가 500 에러)

**V3**: 예매 버튼 클릭 → **5.7ms** 대기 → "대기열 진입!" → polling → 결과 확인

## V3의 한계

| 문제 | 심각도 | 설명 |
|------|--------|------|
| 정합성 | ✅ Solved | 초과 예매, 중복 좌석 완전 해결 |
| DB 보호 | ✅ Solved | 배치 처리로 커넥션 풀 보호 |
| 사용자 경험 | ✅ Improved | 즉시 응답 + 대기열 피드백 |
| Polling 비용 | **Warning** | 6,269 HTTP 요청 (500건의 12배) |
| 이탈 대응 없음 | **High** | 사용자가 대기 중 이탈하면 좌석 영구 점유 |
| TTL 없음 | **High** | 예매 후 미결제 시 좌석 회수 불가 |
| 순번 보장 약함 | **Medium** | 같은 좌석 중복 요청 시 뒤늦게 실패 |

### 이탈 시나리오 (V3의 핵심 문제)

```
1. 사용자 A: 좌석 23번 enqueue → 대기 중 → 브라우저 닫기
2. Scheduler: 좌석 23번 예매 성공 → Redis에 결과 저장
3. 사용자 A: 결과를 polling하지 않음 → 결제도 안 함
4. 좌석 23번: 영구히 점유됨 → 다른 사용자 예매 불가
```

**V4에서 해결**: 입장 토큰 + TTL로 미결제 시 자동 좌석 반환

## 결론

| 항목 | V2 → V3 변화 |
|------|-------------|
| 정합성 | ✅ 유지 |

| 사용자 경험 | ⚠️ → ✅ 즉시 응답 + 대기열 |
| HTTP 에러 | 80% → **0%** |

**V3는 DB를 보호하고 사용자 경험을 개선했지만, 이탈/미결제 처리가 없다.** → V4에서 입장 토큰 + TTL로 해결한다.
