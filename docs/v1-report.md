# V1 — Naive (락 없는 직접 처리) 부하 테스트 리포트

## TL;DR

**100석 콘서트에 500건 동시 예매 → 500건 전부 성공, 좌석당 최대 10명 중복 예매.** `availableSeats`는 50만 감소 (Lost Update). 동시성 제어 없이는 데이터 정합성을 보장할 수 없다.

---

## 테스트 환경

| 항목 | 값 |
|------|-----|
| Backend | Spring Boot 3.4.4, Java 21, HikariCP (pool=10) |
| DB | MySQL 8.0 (Docker, port 3307) |
| 부하 도구 | k6 v0.56.0 |
| VUs | 50 동시 사용자 |
| Iterations | 500 요청 |
| 총 좌석 | 100석 |

## 테스트 시나리오

```
50명의 사용자가 동시에 랜덤 좌석(1~100)으로 예매 요청
→ 총 500건 요청 (좌석 100석)
→ 락 없음, 중복 체크 없음
```

## 원인 분석

### 1. Lost Update (갱신 손실)

```
Thread A: SELECT availableSeats → 100
Thread B: SELECT availableSeats → 100  (같은 값 읽음)
Thread A: UPDATE availableSeats = 99   (100 - 1)
Thread B: UPDATE availableSeats = 99   (100 - 1, A의 갱신을 덮어씀)
→ 2건 예매했지만 1만 차감
```

500건 요청 후 availableSeats가 50이라는 것은, **450건의 갱신이 손실**되었음을 의미한다.

### 2. 중복 좌석 예매 (No Unique Constraint)

- 좌석 번호에 대한 unique constraint 없음
- 동시에 같은 좌석을 예매해도 모두 성공
- 46번 좌석에 10명이 동시에 앉게 되는 상황

### 3. 왜 "정합성이 깨졌는데 에러가 안 나는가"

- 서버는 매 요청마다 `findById` → `availableSeats > 0` 체크 → `save`
- 50개 스레드가 거의 동시에 같은 row를 읽으면, 모두 "좌석 있음"으로 판단
- DB 트랜잭션 격리 수준(REPEATABLE READ)이 있지만, **각 트랜잭션이 시작 시점의 스냅샷**을 봄
- 결과: 모든 요청이 성공으로 응답하지만, 실제 데이터는 엉망

## Little's Law 분석

```
λ (도착률) = 81 req/s
W (평균 처리시간) = 0.575s
L (동시 요청 수) = 81 × 0.575 = 46.6

필요 DB 커넥션 ≈ 47개
실제 Pool Size = 10
→ 부족분 37개 → 대기 발생 → p95 = 2.47초
```

## 결론

| 문제 | 심각도 | 설명 |
|------|--------|------|
| 초과 예매 | **Critical** | 100석에 500건 예매 성공 |
| Lost Update | **Critical** | 500건 중 450건 갱신 손실 |
| 중복 좌석 | **Critical** | 모든 좌석에 다수 중복 |
| 응답 시간 | **Warning** | p95 2.47초 |

**V1은 프로덕션에 절대 사용할 수 없다.** → V2에서 비관적 락으로 정합성 문제를 해결한다.
