# V4 — Token + TTL (입장 토큰) 부하 테스트 리포트

## TL;DR

**100석 콘서트에 500건 동시 예매 → 500건 전부 토큰 발급, 100건 예매 성공, 400건 정상 거절.** 동시 입장 인원을 10명으로 제한하여 DB 부하를 통제하면서, TTL(30초)로 이탈 사용자의 좌석을 자동 회수. HTTP 실패율 0%, 정합성 완벽 유지.

---

## 테스트 환경

| 항목 | 값 |
|------|-----|
| Backend | Spring Boot 3.4.4, Java 21, HikariCP (pool=10) |
| DB | MySQL 8.0 (Docker, port 3307) |
| Redis | Redis 7.0 (Docker, port 6379) |
| 부하 도구 | k6 v0.56.0 |
| VUs | 50 동시 사용자 |
| Iterations | 500 요청 |
| 총 좌석 | 100석 |
| 대기열 | Redis Sorted Set |
| 토큰 | Redis String + TTL 30초 |
| 동시 입장 제한 | 최대 10명 |

## 테스트 시나리오

```
50명의 사용자가 동시에 랜덤 좌석(1~100)으로 예매 요청
→ Redis 대기열에 enqueue (즉시 응답)
→ Scheduler가 200ms마다 배치 dequeue → 토큰 발급 (최대 10개 동시)
→ 클라이언트가 polling으로 토큰 수신
→ 토큰으로 좌석 예매 (비관적 락)
→ 성공 시 토큰 소멸, 실패 시 토큰 소멸 + 다음 대기자 입장
→ TTL(30초) 만료 시 자동 토큰 삭제 → 다음 대기자 입장
```

### 토큰 흐름

```
1. 사용자 → enqueue → 대기열 진입 (rank 부여)
2. Scheduler → dequeue → 토큰 발급 (최대 10개 동시)
3. 사용자 ← polling ← "TOKEN_ISSUED" + token UUID
4. 사용자 → reserve-with-token → 토큰 검증 → 비관적 락 → 예매
5a. 성공 → 토큰 소멸 → active count 감소 → 다음 대기자 입장
5b. 실패 → 토큰 소멸 → active count 감소 → 다음 대기자 입장
5c. 이탈 → TTL 30초 후 자동 소멸 → active count 감소 → 다음 대기자 입장
```

### 핵심 설정값과 산출 근거

```
Token TTL = 30초

산출: W_reservation(좌석 선택 + 결제) × safety(2x) + UX_buffer
= 10초 × 2 + 10초 = 30초

Max Active Tokens = 10

산출: DB Pool Size × 안전 계수
= 10 × 1.0 = 10
(토큰 보유자가 언제든 DB 접근할 수 있으므로, pool과 동일하게 제한)

Scheduler Batch Size = min(5, 10 - activeTokens)
→ 활성 토큰이 적으면 빠르게 채움
→ 활성 토큰이 꽉 차면 발급 중단

Little's Law (토큰 기반):
λ_token = activeTokens / W_reservation = 10 / 0.05 = 200 req/s (이론상 최대)
실제: 토큰 보유자가 순차적으로 예매하므로 ~15-20 req/s
L_db = 동시 DB 트랜잭션 ≤ 10 (토큰 수로 hard limit)
```

## V3 vs V4: 결정적 차이

### 이탈 처리

**V3 (대기열만)**
```
사용자 A: enqueue → Scheduler가 예매 → 성공
사용자 A: 브라우저 닫기 → polling 중단
→ 좌석은 이미 예매됨 → 결제 안 함 → 좌석 영구 점유
```

**V4 (토큰 + TTL)**
```
사용자 A: enqueue → 토큰 발급 → 브라우저 닫기
→ 30초 후 토큰 TTL 만료 → Redis에서 자동 삭제
→ active count 감소 → 다음 대기자에게 토큰 발급
→ 좌석은 아직 점유되지 않음 (토큰 보유 ≠ 좌석 점유)
```

### Polling 효율

| 지표 | V3 | V4 |
|------|-----|-----|
| Polling 요청 수 | 5,765 (11.5배) | **3,520 (7배)** |
| 이유 | 스케줄러가 예매까지 완료해야 결과 나옴 | 토큰만 발급하면 되므로 빠름 |

### 동시 DB 접근 제어

**V3**: 배치 5건 → 스케줄러가 예매까지 수행 → DB에 5건 동시  
**V4**: 토큰 10개 → 사용자가 개별적으로 예매 → DB에 최대 10건이지만, 사용자 행동 시간 분산으로 실제 2-3건

## TTL 만료 시나리오 테스트

> **참고**: 이 부하 테스트에서는 모든 사용자가 토큰을 즉시 사용하므로 TTL 만료가 발생하지 않았다.
> TTL 효과를 증명하려면 별도의 이탈 시뮬레이션 테스트가 필요하다.

### 기대 동작

```
1. 사용자 100명이 대기열 진입
2. 토큰 10개 발급 (동시 입장 10명)
3. 3명이 이탈 (토큰 미사용)
4. 30초 후 → TTL 만료 → active count 7로 감소
5. Scheduler → 3개 토큰 추가 발급 → active count 다시 10
6. 새로운 3명이 입장하여 예매 진행
```

## 결론

V4는 V1~V3의 모든 문제를 해결한 최종 아키텍처:

1. **V1 문제 (정합성)** → V2에서 비관적 락으로 해결
2. **V2 문제 (DB 과부하)** → V3에서 Redis 대기열로 해결
3. **V3 문제 (이탈 미대응)** → V4에서 토큰 TTL로 해결

**프로덕션 적용 시 추가 고려사항:**
- Token TTL은 실제 결제 소요시간에 맞게 조정 (30초 → 5분 등)
- Max Active Tokens는 DB 커넥션 풀과 연동하여 동적 조정
- WebSocket 또는 SSE로 polling 대체 (HTTP 요청 수 감소)
- 분산 환경에서 Redis Cluster 구성
